# Commit Workflow

## Context

Current branch:
!`git branch --show-current`

Staged changes:
!`git diff --cached --stat 2>/dev/null || echo "Nothing staged"`

Unstaged changes:
!`git diff --stat`

Untracked files:
!`git ls-files --others --exclude-standard | head -20`

Recent commits (for style reference):
!`git log --oneline -5`

---

## Phase 0: Branch Safety Check

**CRITICAL: Verify you are NOT on master/main before committing.**

```bash
CURRENT_BRANCH=$(git branch --show-current)
DEFAULT_BRANCH=$(git symbolic-ref refs/remotes/origin/HEAD 2>/dev/null | sed 's@^refs/remotes/origin/@@' || echo "main")

if [ "$CURRENT_BRANCH" = "$DEFAULT_BRANCH" ] || [ "$CURRENT_BRANCH" = "master" ] || [ "$CURRENT_BRANCH" = "main" ]; then
  echo "⛔ ERROR: You are on $CURRENT_BRANCH. Create a feature branch first!"
  echo "Run: git checkout -b <feature-branch-name>"
  exit 1
fi
```

**If on master/main:** STOP. Create a feature branch first. NEVER commit directly to the default branch.

---

## Phase 1: Review Changes

1. **Examine the full diff**
   ```bash
   git diff          # unstaged
   git diff --cached # staged
   ```

2. **Understand the scope**
   - What feature/fix/refactor is this?
   - Is this one logical unit of work or should it be multiple commits?
   - Are there any changes that shouldn't be committed (debug code, console.logs, TODOs)?

3. **Stage the changes**
   ```bash
   git add <specific-files>      # preferred: stage specific files
   ```

   **WARNING:** Avoid `git add -A` or `git add .` — these can accidentally stage sensitive files (`.env`, credentials, large binaries). Always review with `git diff --name-only` first and stage specific files.

## Phase 2: Craft Commit Message

**Format: Conventional Commits**

```
<type>(<scope>): <short description>

<body - what and why, not how>

<footer - references, breaking changes>
```

### Types
| Type | Use for |
|------|---------|
| `fix` | Bug fixes |
| `feat` | New features |
| `refactor` | Code restructuring (no behavior change) |
| `perf` | Performance improvements |
| `docs` | Documentation only |
| `test` | Adding/updating tests |
| `chore` | Build, deps, config changes |
| `style` | Formatting, whitespace (no code change) |

### Rules

**Subject line:**
- Lowercase, no period at end
- Imperative mood ("add" not "added" or "adds")
- Max 50 characters ideally, 72 hard limit
- Be specific: "fix null crash in position sync" not "fix bug"

**Body (if needed):**
- Wrap at 72 characters
- Explain *what* changed and *why*, not *how*
- Reference related issues/context

**Examples:**

Simple fix:
```
fix(grid-bot): handle empty position array on startup
```

With body:
```
feat(hedge-engine): add volatility regime detection

Implements MA-based volatility classification to adjust hedge
ratios dynamically. Uses 20/50 period crossover to identify
high/low volatility regimes.

Relates to #45
```

Breaking change:
```
refactor(api)!: rename position endpoints

BREAKING CHANGE: /positions/active renamed to /positions/open
```

**IMPORTANT: DO NOT include any of the following:**
- Co-Authored-By lines
- "Generated by Claude" or similar attribution
- Any reference to Claude, Anthropic, or AI assistance

## Phase 3: Confirmation

**Present the commit for review:**

```
📝 Proposed Commit
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

<type>(<scope>): <subject>

<body>

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Files: <list of files>
Branch: <current branch>
```

**⏸️ Wait for confirmation before executing `git commit`**

---

## Phase 4: Commit

Once confirmed:
```bash
git commit -m "<message>"
```

Or for multi-line:
```bash
git commit -m "<subject>" -m "<body>"
```

---

## Quick Reference

```
# Undo last commit (keep changes) — use with caution
git reset --soft HEAD~1

# Amend last commit (only if NOT yet pushed)
git commit --amend
```

> **Note:** Interactive commands like `git add -p`, `git add -i`, and `git rebase -i` require TTY input and are not supported in this environment.
