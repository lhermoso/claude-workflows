# Quick Fix - Autonomous Issue Resolution

## Context

Issue to fix:
!`gh issue view $ARGUMENTS --json number,title,body,labels`

Repository:
!`git remote get-url origin`

---

## Execution Mode: AUTONOMOUS

This command runs with **minimal human intervention**. Only stops if:
- Ambiguity that could lead to wrong solution
- Tests failing after 2 retry attempts
- Critical error blocking progress

---

## Step 1: Setup Worktree (Silent)

```bash
DEFAULT_BRANCH=$(git symbolic-ref refs/remotes/origin/HEAD 2>/dev/null | sed 's@^refs/remotes/origin/@@' || echo "main")
git fetch origin $DEFAULT_BRANCH

# Handle existing worktree directory
if [ -d "../quick-fix-$ARGUMENTS" ]; then
  cd ../quick-fix-$ARGUMENTS
else
  git worktree add ../quick-fix-$ARGUMENTS -b fix/$ARGUMENTS origin/$DEFAULT_BRANCH
  cd ../quick-fix-$ARGUMENTS
fi
```

---

## Step 2: Plan (MANDATORY)

**Before implementing, you MUST switch to plan mode:**

```
Use the EnterPlanMode tool to switch to plan mode before creating your implementation plan.
```

1. Read issue #$ARGUMENTS completely
2. Search codebase for relevant files
3. **Identify ROOT CAUSE** — do NOT apply surface-level fixes (z-index, overflow-hidden, retry loops). Ask "why does this happen?" until you reach the actual cause.
   - For UI bugs: find the actual layout/box model issue before reaching for CSS hacks
   - For numerical bugs: verify input parameters and config values first
   - For runtime errors: trace the data flow to find where it breaks
4. Create a brief implementation plan

**Use the ExitPlanMode tool to get user approval before implementing.**

---

## Step 3: Implement (Test-First, Autonomous)

1. **Write a failing test first** that reproduces the bug
2. Implement the minimal fix
3. Verify the test now passes
4. Consider edge cases (at least 2-3 related scenarios)

**Decision Guidelines:**
- If multiple approaches exist, pick the simplest one
- If unclear, check similar past PRs for patterns
- Keep changes minimal and focused
- **NEVER apply surface-level fixes** without understanding root cause

---

## Step 4: Verify (Auto-retry)

```bash
# Run tests (detect project type)
if [ -f "package.json" ]; then
  npm test
elif [ -f "Cargo.toml" ]; then
  cargo test
elif [ -f "pyproject.toml" ] || [ -f "setup.py" ]; then
  pytest
fi
```

If tests fail:
1. Analyze failure
2. Fix the issue
3. Retry (max 2 attempts)
4. If still failing after 2 attempts, report and stop

---

## Step 5: Update Changelog (if exists)

**If the project has a CHANGELOG.md, update it before committing:**

```bash
# Check if CHANGELOG.md exists
ls CHANGELOG.md 2>/dev/null
# If yes, add entry under [Unreleased] → Fixed/Added/Changed section
```

> If no CHANGELOG.md exists, skip this step.

## Step 6: Commit & Push (Auto)

```bash
# Stage specific files (avoid git add -A to prevent staging secrets/binaries)
git add <specific-files>

git commit -m "fix: <auto-generated description>

Fixes #$ARGUMENTS"
git push -u origin HEAD
```

**IMPORTANT: Never include Co-Authored-By lines or any Claude/Anthropic attribution**

---

## Step 7: Create PR (Auto)

```bash
gh pr create \
  --title "fix: <description from issue>" \
  --body "## Summary
Fix for #$ARGUMENTS

## Changes
<auto-generated from diff>

## Changelog
- [x] CHANGELOG.md updated

## Testing
- [x] Tests pass"
```

---

## Step 8: Self-Review

**Perform a quick review of the changes:**

1. Check the PR diff: `gh pr diff <pr-number>`
2. Verify: no debug code, no secrets, no breaking changes, tests pass
3. Check changelog was updated (if applicable)

> **Note:** You cannot approve your own PR on GitHub. If the repo requires approval, leave a comment instead:
> ```bash
> gh pr comment <pr-number> --body "Self-review passed: no breaking changes, no debug code, tests pass"
> ```
> If the repo allows merge without approval, proceed directly to merge.
