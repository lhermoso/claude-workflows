# Batch Issue Processor

## Context

Repository:
!`git remote get-url origin`

Open issues:
!`gh issue list --state open --limit 20`

---

## Input Parsing

Based on input: **$ARGUMENTS**

Parse the input to get issue list:

| Input Format | Example | Action |
|--------------|---------|--------|
| Comma-separated numbers | `12,15,18` | Process issues #12, #15, #18 |
| Range | `10-15` | Process issues #10 through #15 |
| Label filter | `label:bug` | Get all issues with "bug" label |
| Assignee filter | `assignee:@me` | Get issues assigned to you |
| `all-open` | `all-open` | Process all open issues (careful!) |
| Search query | `"authentication"` | Search issues by keyword |

---

## Phase 1: Gather Issue List

```bash
# For label filter
gh issue list --label "<label>" --state open --json number,title

# For assignee
gh issue list --assignee "@me" --state open --json number,title

# For all open
gh issue list --state open --json number,title

# For search
gh issue list --search "<query>" --state open --json number,title
```

**Present the list for confirmation:**

```
Found X issues to process:

#12 - Fix login timeout error
#15 - Add dark mode toggle
#18 - Update API rate limiting

Proceed with all? (Enter to continue, or specify subset)
```

---

## Phase 2: Parallel Processing with Subagents

For each issue, launch an independent subagent:

```
For issue #12, #15, #18 - Launch 3 parallel Task agents:

Task 1: "Process issue #12 end-to-end:
- Detect default branch: git symbolic-ref refs/remotes/origin/HEAD 2>/dev/null | sed 's@^refs/remotes/origin/@@' || echo 'main'
- Create worktree ../fix-12-<desc> from origin/<default-branch>
- Fetch full issue with comments: `gh issue view 12 --json number,title,body,labels,comments`
- Read the issue body AND all comments — comments often contain reproduction steps, clarifications, or constraints
- Identify ROOT CAUSE (not surface-level symptoms)
- Write a failing test that reproduces the bug
- Create a brief implementation plan
- Implement fix with minimal changes
- Run tests (new test should pass, full suite should pass)
- Update CHANGELOG.md if one exists (add entry under [Unreleased])
- Stage specific files (never use git add -A)
- Create commit (NO Co-Authored-By or Claude attribution)
- Create PR (NO 'Generated by Claude' text)
- Self-review the changes
Return: {issue: 12, pr: <number>, status: success|failed, notes: <any issues>}"

Task 2: "Process issue #15 end-to-end..."

Task 3: "Process issue #18 end-to-end..."
```

**Key points:**
- Each agent works in its own worktree (no conflicts)
- Agents run in parallel (major time savings)
- Each returns structured results

While agents work, periodically check:

## Phase 5: Batch Review (Optional)

If requested, review each PR:

```
For each PR created:
  1. Get diff: gh pr diff <pr-number>
  2. Check: no debug code, no secrets, no breaking changes, tests pass, root cause addressed (not surface-level fix)
  3. Verdict: SAFE TO MERGE or NEEDS CHANGES
  4. Leave COMMENT review on PR (NEVER use --approve or --request-changes on your own PRs)
  5. For non-blocking findings, create GitHub issues to track them
```

> **IMPORTANT:** You cannot approve/request-changes on your own PRs on GitHub. Always use `gh pr review <number> --comment --body "..."` for self-authored PRs.

**Merge preference:** Use `gh pr merge <number> --rebase --delete-branch`. Fall back to `--squash` if rebase fails.

## Cleanup Helper

After PRs are merged:

```bash
# Remove all fix worktrees
git worktree list | grep "fix-" | awk '{print $1}' | xargs -I {} git worktree remove {}

# Prune references
git worktree prune

# Delete merged branches
git branch --merged main | grep "fix/" | xargs git branch -d
```

---

## Safety Limits

- **Max parallel agents:** 3 (to avoid overwhelming the system and context overflow)
- **Confirmation required:** Always show list before processing
- **Auto-stop on critical failure:** If 3+ consecutive failures, pause and report
